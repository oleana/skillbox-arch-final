# 12. Сценарии использования приложения

# Назначение документа

Документ описывает основные сценарии использования приложения.  
Сценарии используются для проверки того, что архитектура покрывает ключевые бизнес-требования и пользовательские действия.

---

# Основные акторы

| Актор | Описание |
|------|----------|
| Пользователь | Спортсмен, который тренируется, анализирует результаты, общается и участвует в соревнованиях |
| Маркетинг | Настраивает акции, новости и персональные предложения |
| Администратор | Управляет платформой, пользователями и настройками |

# UC-01. Регистрация и вход

## Цель

Дать пользователю доступ к приложению.

## Основной поток

1. Пользователь открывает приложение.
2. Выбирает регистрацию или вход.
3. Identity Service проверяет данные.
4. Система создаёт или находит профиль.
5. Пользователь получает JWT.
6. Открывается главный экран.

## Задействованные сервисы

- API Gateway
- Identity Service
- User Profile Service
- User DB

---

# UC-02. Запись тренировки

## Цель

Сохранить тренировку пользователя и показатели с устройств.

## Основной поток

1. Пользователь запускает тренировку.
2. Training Service создаёт тренировку.
3. Integration Service получает данные со смартфона и устройств.
4. Во время тренировки показатели сохраняются.
5. После завершения тренировки итоговые результаты записываются в БД.
6. В Kafka публикуется событие `WorkoutCompleted`.

## Задействованные сервисы

- API Gateway
- Training Service
- Integration Service
- Kafka
- Training DB

---

# UC-03. Анализ результатов

## Цель

Показать пользователю прогресс и сравнение с предыдущими тренировками.

## Основной поток

1. Пользователь открывает историю тренировок.
2. Training Service получает данные тренировок.
3. Analytics Service рассчитывает агрегаты.
4. Система показывает прогресс.
5. Дополнительно выполняется сравнение с пользователями региона.

## Задействованные сервисы

- Training Service
- Analytics Service
- Data Lake
- Training DB

---

# UC-04. Поиск единомышленников

## Цель

Помочь пользователю найти людей для совместных тренировок.

## Основной поток

1. Пользователь задаёт критерии поиска.
2. Search Service ищет пользователей по интересам, маршрутам или региону.
3. Social Service возвращает группы и связи.
4. Пользователь просматривает найденных людей.
5. Пользователь отправляет приглашение.

## Задействованные сервисы

- Search Service
- Social Service
- Redis
- Social DB

---

# UC-05. Получение рекомендаций

## Цель

Предложить пользователю тренировку, инвентарь или промоакцию.

## Основной поток

1. Recommendation Service получает данные пользователя.
2. Analytics Service предоставляет историю и агрегаты.
3. Equipment Service передаёт состояние инвентаря.
4. Promotion Service передаёт доступные акции.
5. Пользователь получает персональные рекомендации.

## Задействованные сервисы

- Recommendation Service
- Analytics Service
- Equipment Service
- Promotion Service
- Data Lake
- Redis

---

# UC-06. Покупка спортивного товара

## Цель

Перевести пользователя из рекомендации к покупке товара.

## Основной поток

1. Пользователь получает рекомендацию товара.
2. Система показывает персональную или региональную акцию.
3. Пользователь переходит в интернет-магазин компании.
4. Покупка оформляется в существующей e-commerce системе.

## Задействованные сервисы

- Recommendation Service
- Promotion Service
- Интернет-магазин компании

---

# UC-07. Участие в соревновании

## Цель

Обеспечить участие большого количества пользователей в соревновании.

## Основной поток

1. Пользователь вступает в соревнование.
2. Во время тренировки система принимает события.
3. Training Service сохраняет результаты.
4. Kafka передаёт события в Gamification Service и Analytics Service.
5. Gamification Service обновляет рейтинг.
6. Notification Service отправляет уведомления.
7. Пользователь видит обновлённые результаты.

## Задействованные сервисы

- Training Service
- Kafka
- Gamification Service
- Analytics Service
- Notification Service
- Redis

---

# UC-08. Управление акциями и новостями

## Цель

Позволить маркетингу управлять промоакциями и спортивными новостями.

## Основной поток

1. Маркетинг входит в Admin Portal.
2. API Gateway проверяет JWT и роль.
3. Promotion Service создаёт или изменяет акцию.
4. Акция сохраняется в Promotion DB.
5. Пользователи получают релевантные предложения.

## Задействованные сервисы

- Admin Portal
- API Gateway
- Identity Service
- Promotion Service
- Promotion DB

---

# UC-09. Администрирование платформы

## Цель

Позволить администратору управлять платформой и контролировать её состояние.

## Основной поток

1. Администратор входит в Admin Portal.
2. API Gateway проверяет JWT и роль администратора.
3. Администратор управляет настройками платформы.
4. Действия администратора записываются в аудит.

## Задействованные сервисы

- Admin Portal
- API Gateway
- Identity Service
- User Profile Service
- Observability / Audit
