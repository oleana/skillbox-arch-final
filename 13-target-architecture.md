# 13. Детальная архитектура

# Назначение документа

Документ описывает базовую архитектуру с учётом ограничений бизнес-требований, НФТ, выбранной архитектуры, адресация атрибутов качества.

Архитектура построена с учётом требований к масштабируемости, высокой доступности, безопасности и возможности дальнейшего развития продукта.

---

# Общая структура

Платформа реализована в виде набора независимых микросервисов, взаимодействующих посредством REST API и событийной шины Kafka.

Основные группы компонентов:

- клиентские приложения;
- API-слой;
- доменные микросервисы;
- инфраструктурные сервисы;
- системы хранения данных;
- внешние системы.

---

# Клиентский уровень

Клиентский уровень включает:

- Mobile App;
- Admin Portal.

Все пользовательские запросы поступают через **Global Load Balancer**, после чего направляются в ближайший регион.

Статический контент доставляется через CDN.

---

# API-уровень

Единственной точкой входа является **API Gateway**.

Основные функции:

- маршрутизация запросов;
- проверка JWT;
- rate limiting;
- базовая авторизация;
- логирование;
- передача запросов в соответствующие микросервисы.

---

# Доменные сервисы

Сервисы разделены по бизнес-доменам.

| Домен | Назначение |
|--------|------------|
| Identity | Аутентификация, выпуск JWT, управление ролями |
| User Profile | Профили пользователей и настройки приватности |
| Training | Запись и хранение тренировок |
| Social | Группы, друзья, комментарии |
| Search | Поиск пользователей и маршрутов |
| Recommendation | Персональные рекомендации |
| Equipment | Учёт спортивного инвентаря |
| Gamification | Достижения, рейтинги, соревнования |
| Notification | Push-уведомления |
| Promotion | Акции и новости |
| Analytics | Анализ тренировок |
| Integration | Интеграция с внешними системами |

Каждый сервис развивается независимо и владеет собственными данными.

---

# Хранение данных

Используется паттерн **Database per Service**.

Каждый сервис имеет собственную базу данных и не обращается напрямую к данным других сервисов.

Обмен информацией выполняется через:

- REST API;
- Kafka.

Для хранения различных типов данных используются:

- PostgreSQL;
- TimescaleDB (тренировочные данные);
- Redis;
- Object Storage;
- Data Lake.

---

# Межсервисное взаимодействие

Используются два варианта взаимодействия.

## Синхронное

REST over HTTPS (mTLS).

Используется для:

- пользовательских запросов;
- получения данных между сервисами;
- административных операций.

---

## Асинхронное

Kafka.

Используется для:

- событий тренировок;
- достижений;
- аналитики;
- рекомендаций;
- уведомлений;
- интеграций.

---

# Безопасность

За безопасность отвечают следующие компоненты:

| Компонент | Назначение |
|-----------|------------|
| Identity Service | Аутентификация пользователей |
| API Gateway | Проверка JWT, базовая авторизация |
| Доменные сервисы | Проверка бизнес-прав доступа |
| mTLS | Аутентификация сервисов |
| RBAC | Разграничение административного доступа |

Пользователь проходит аутентификацию один раз, после чего все запросы выполняются с использованием JWT.

---

# Масштабирование

Все сервисы являются stateless.

Горизонтально масштабируются:

- API Gateway;
- Training Service;
- Search Service;
- Notification Service;
- Analytics Service;
- Recommendation Service.

Масштабирование выполняется Kubernetes Horizontal Pod Autoscaler.

---

# Отказоустойчивость

Для обеспечения высокой доступности используются:

- несколько экземпляров каждого критичного сервиса;
- репликация баз данных;
- Kafka Cluster;
- Redis Cluster;
- автоматический restart контейнеров;
- rolling update;
- health checks;
- readiness и liveness probes.

---

# Глобальная доступность

Платформа проектируется как мультирегиональная.

Основные компоненты:

- Global Load Balancer;
- CDN;
- Kubernetes-кластеры в нескольких регионах;
- репликация критичных данных;
- автоматическое переключение между регионами.

Такой подход обеспечивает минимальные задержки для пользователей и устойчивость к отказу отдельного региона.

---

# Наблюдаемость

Для эксплуатации используются:

- Prometheus;
- Grafana;
- Loki;
- Tempo;
- Alertmanager.

Собираются:

- технические метрики;
- бизнес-метрики;
- централизованные логи;
- распределённые трассировки.

---

# Архитектурные принципы

При разработке платформы используются следующие принципы:

- Domain Driven Design;
- Database per Service;
- API First;
- Event Driven Architecture;
- Stateless Services;
- Zero Trust;
- Infrastructure as Code.

---

# Вывод

Целевая архитектура обеспечивает независимое развитие бизнес-доменов, горизонтальное масштабирование, безопасное взаимодействие компонентов и поддержку пользователей по всему миру. Выбранные архитектурные решения позволяют постепенно расширять функциональность платформы без изменения её базовой структуры.
