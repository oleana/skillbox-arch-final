# 7. Критические бизнес-сценарии

# Назначение документа

Документ описывает наиболее важные бизнес-сценарии системы, от успешного выполнения которых зависит достижение бизнес-целей проекта.

При проектировании архитектуры именно эти сценарии рассматриваются как приоритетные. Все архитектурные решения должны обеспечивать их надёжное выполнение даже при высокой нагрузке и частичном отказе отдельных компонентов.

---

# BS-01. Регистрация и авторизация пользователя

## Описание

Новый пользователь устанавливает приложение, регистрируется и получает доступ ко всем функциям платформы.

## Участники

- Пользователь
- Identity Service
- User Profile Service

## Основной сценарий

1. Пользователь открывает приложение.
2. Выполняет регистрацию или вход.
3. Система проверяет учётные данные.
4. Создаётся профиль пользователя.
5. Пользователь получает JWT-токен.
6. Открывается главный экран приложения.

## Критичность

Очень высокая.

Без выполнения данного сценария использование приложения невозможно.

---

# BS-02. Запись тренировки

## Описание

Пользователь начинает тренировку, подключает спортивное устройство и сохраняет результаты.

## Участники

- Пользователь
- Training Service
- Integration Service

## Основной сценарий

1. Пользователь запускает тренировку.
2. Система получает данные со смартфона и подключённых устройств.
3. Во время тренировки показатели сохраняются.
4. После завершения тренировки результаты записываются в систему.

## Критичность

Очень высокая.

Это основной пользовательский сценарий платформы.

---

# BS-03. Анализ результатов

## Описание

После завершения тренировки пользователь анализирует полученные результаты.

## Участники

- Training Service
- Analytics Service

## Основной сценарий

1. Пользователь открывает историю тренировок.
2. Система рассчитывает показатели.
3. Отображается прогресс относительно предыдущих тренировок.
4. Выполняется сравнение с пользователями региона.

## Критичность

Высокая.

Именно анализ результатов мотивирует пользователя продолжать пользоваться приложением.

---

# BS-04. Поиск единомышленников

## Описание

Пользователь хочет найти людей с похожими интересами для совместных тренировок.

## Участники

- Search Service
- Social Service

## Основной сценарий

1. Пользователь задаёт критерии поиска.
2. Система ищет пользователей.
3. Отображаются найденные группы и спортсмены.
4. Пользователь отправляет приглашение.

## Критичность

Высокая.

Социальное взаимодействие является одним из основных инструментов повышения вовлечённости пользователей.

---

# BS-05. Получение персональных рекомендаций

## Описание

После накопления истории тренировок пользователь получает персональные рекомендации.

## Участники

- Recommendation Service
- Analytics Service

## Основной сценарий

1. Аналитическая система обрабатывает историю тренировок.
2. Формируются рекомендации.
3. Пользователь получает советы по тренировкам.
4. При необходимости предлагается обновить спортивный инвентарь.

## Критичность

Средняя.

Повышает ценность приложения и увеличивает вероятность регулярного использования.

---

# BS-06. Покупка спортивных товаров

## Описание

Пользователь получает рекомендацию по покупке спортивного инвентаря.

## Участники

- Recommendation Service
- Promotion Service
- Интернет-магазин компании

## Основной сценарий

1. Пользователь получает персональное предложение.
2. Переходит в интернет-магазин.
3. Покупает рекомендованный товар.

## Критичность

Высокая.

Данный сценарий напрямую связан с достижением одной из ключевых бизнес-целей — увеличением продаж продукции компании.

---

# BS-07. Массовое соревнование

## Описание

Одновременно большое количество пользователей участвует в соревновании.

## Участники

Практически все сервисы платформы.

## Основной сценарий

1. Пользователи начинают соревнование.
2. Система принимает большое количество событий.
3. Выполняется обработка результатов.
4. Обновляются рейтинги.
5. Пользователи получают уведомления.

## Критичность

Очень высокая.

Этот сценарий определяет требования к масштабируемости всей платформы.

# Влияние на архитектуру

Критические бизнес-сценарии определили следующие архитектурные решения:

- использование API Gateway в качестве единой точки входа;
- выделение отдельных микросервисов по бизнес-доменам;
- применение событийной архитектуры для обработки тренировок и уведомлений;
- горизонтальное масштабирование сервисов;
- использование Kafka для обработки большого количества событий;
- внедрение централизованного мониторинга и трассировки.
