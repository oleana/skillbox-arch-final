# 4. Концептуальная архитектура

## Назначение документа

Документ описывает концептуальную архитектуру решения и основные архитектурные принципы. На данном этапе определяется общий подход к построению системы без детальной проработки отдельных компонентов.

---

# Архитектурный подход

Для реализации проекта выбрана микросервисная архитектура.

Такой подход позволяет независимо разрабатывать, масштабировать и сопровождать отдельные функциональные области системы. Кроме того, архитектура хорошо подходит для глобального приложения с большим количеством пользователей, высокой нагрузкой и большим числом команд разработки.

Взаимодействие между сервисами строится по двум моделям:

- синхронное взаимодействие (REST/gRPC);
- асинхронный обмен событиями через Kafka.

Подобная схема позволяет использовать преимущества каждого подхода: минимальные задержки для пользовательских операций и слабую связанность сервисов для фоновых процессов.

---

# Основные архитектурные принципы

При разработке системы используются следующие принципы.

- разделение системы по бизнес-доменам;
- независимое развертывание сервисов;
- Stateless-сервисы;
- API First;
- Event Driven Architecture;
- Security by Design;
- Observability by Design;
- Infrastructure as Code.

---

# Основные домены системы

Для уменьшения связанности система разделена на несколько независимых доменов.

| Домен | Назначение |
|--------|------------|
| Identity | Аутентификация и авторизация пользователей |
| User Profile | Управление профилем пользователя |
| Training | Управление тренировками |
| Social | Социальные функции |
| Recommendation | Персональные рекомендации |
| Equipment | Учёт спортивного инвентаря |
| Notification | Отправка уведомлений |
| Promotion | Персональные акции |
| Analytics | Анализ тренировок |
| Integration | Интеграция с внешними системами |

Каждый домен развивается независимо и имеет собственную базу данных.

---

# Взаимодействие компонентов

Пользовательские приложения взаимодействуют только с API Gateway.

API Gateway выполняет:

- аутентификацию запросов;
- маршрутизацию;
- ограничение частоты запросов;
- агрегацию ответов;
- ведение журналов аудита.

После прохождения Gateway запрос направляется в соответствующий микросервис.

Для пользовательских операций используется REST API.

Для внутренних высоконагруженных процессов используется обмен событиями через Kafka.

---

# Работа с данными

Каждый микросервис владеет собственной базой данных.

Совместное использование одной базы данных несколькими сервисами не допускается.

Обмен данными осуществляется только через API либо публикацию событий.

Такой подход обеспечивает:

- независимое развитие сервисов;
- отсутствие сильной связанности;
- возможность независимого масштабирования.

---

# Масштабирование

Система проектируется с возможностью горизонтального масштабирования.

Все сервисы являются stateless и могут запускаться в нескольких экземплярах.

Масштабирование выполняется автоматически средствами Kubernetes.

Для хранения пользовательских данных используются отказоустойчивые кластеры баз данных.

---

# Отказоустойчивость

При проектировании учитываются следующие требования.

- отказ одного сервиса не должен приводить к отказу всей системы;
- внешние интеграции должны иметь механизм graceful degradation;
- длительные операции выполняются асинхронно;
- сервисы автоматически восстанавливаются средствами Kubernetes.

---

# Безопасность

Основные принципы безопасности.

- OAuth2/OpenID Connect;
- JWT;
- mTLS между сервисами;
- шифрование персональных данных;
- централизованный аудит;
- разграничение доступа по ролям.

---

# Наблюдаемость

Для обеспечения эксплуатации используются:

- централизованное логирование;
- сбор метрик;
- распределённая трассировка;
- мониторинг бизнес-метрик.

Это позволяет быстро обнаруживать и локализовывать возникающие проблемы.

## Принятые решения

| Решение | Причина |
|----------|----------|
| Микросервисная архитектура | Независимое развитие сервисов |
| Kubernetes | Горизонтальное масштабирование |
| API Gateway | Единая точка входа |
| Kafka | Асинхронное взаимодействие |
| PostgreSQL | Надёжное хранение транзакционных данных |
| Redis | Кэширование |
| Object Storage | Хранение пользовательского контента |
