# 9. Нефункциональные требования

# Назначение документа

Документ описывает нефункциональные требования к системе.

НФТ сформированы на основе бизнес-целей, критических бизнес-сценариев и выбранных атрибутов качества: масштабируемость, доступность и безопасность.

---

# NFR-01. Доступность

Система должна обеспечивать высокую доступность основных пользовательских сценариев.

| Требование | Значение |
|-----------|----------|
| Доступность критичных функций | 99,95% |
| Доступность некритичных функций | 99,5% |
| RTO для критичных сервисов | до 15 минут |
| RPO для критичных данных | до 5 минут |

Критичные функции:

- регистрация и авторизация;
- запуск и завершение тренировки;
- сохранение результатов тренировки;
- просмотр истории тренировок.

---

# NFR-02. Производительность

Система должна обеспечивать быстрый отклик пользовательского интерфейса и API.

| Операция | Требование |
|----------|------------|
| Авторизация | до 2 секунд |
| Получение профиля | P95 ≤ 300 мс |
| Запись показателей тренировки | P95 ≤ 500 мс |
| Получение истории тренировок | P95 ≤ 700 мс |
| Поиск пользователей | P95 ≤ 1 сек |
| Получение рекомендаций | P95 ≤ 1,5 сек |

Для тяжёлых операций допускается асинхронная обработка.

---

# NFR-03. Масштабируемость

Система должна поддерживать горизонтальное масштабирование.

Требования:

- stateless-сервисы;
- автоматическое масштабирование в Kubernetes;
- возможность увеличения количества экземпляров сервисов без остановки системы;
- отдельное масштабирование сервисов тренировок, уведомлений, аналитики и соревнований;
- поддержка пиковых нагрузок во время массовых соревнований.

---

# NFR-04. Безопасность

Система должна обеспечивать защиту персональных данных пользователей и данных тренировок.

Требования:

- OAuth2 / OpenID Connect;
- JWT для пользовательских сессий;
- mTLS для межсервисного взаимодействия;
- RBAC для доступа к административным функциям;
- шифрование персональных данных;
- шифрование данных при передаче;
- аудит действий пользователей и администраторов;
- защита публичного API через API Gateway и Rate Limiting.

---

# NFR-05. Конфиденциальность данных

Система хранит чувствительные данные:

- персональные данные пользователя;
- геолокацию;
- маршруты тренировок;
- спортивные показатели;
- данные с внешних устройств;
- информацию о здоровье, например пульс и уровень кислорода.

Требования:

- пользователь должен управлять настройками приватности;
- данные о маршрутах не должны быть публичными по умолчанию;
- пользователь должен явно разрешать публикацию достижений;
- данные должны обрабатываться с учётом GDPR;
- для аналитики должны использоваться агрегированные или обезличенные данные.

---

# NFR-06. Наблюдаемость

Система должна быть наблюдаемой на уровне логов, метрик и трассировки.

Требования:

- централизованный сбор логов;
- сбор технических и бизнес-метрик;
- распределённая трассировка запросов;
- алерты по критичным сценариям;
- dashboards для DevOps, разработки и бизнеса.

Ключевые метрики:

- количество активных пользователей;
- количество начатых и завершённых тренировок;
- процент ошибок API;
- latency по ключевым операциям;
- lag Kafka;
- количество неотправленных уведомлений;
- нагрузка на сервисы и базы данных.

---

# NFR-07. Отказоустойчивость

Отказ отдельного сервиса не должен приводить к отказу всей платформы.

Требования:

- retry с ограничением количества попыток;
- timeout для внешних вызовов;
- circuit breaker для нестабильных интеграций;
- graceful degradation для некритичных функций;
- автоматический restart сервисов;
- репликация критичных баз данных;
- резервное копирование данных.

Пример: если недоступен Recommendation Service, пользователь всё равно должен иметь возможность записать тренировку.

---

# NFR-08. Интеграции

Система должна безопасно интегрироваться с внешними сервисами.

Интеграции:

- интернет-магазин компании;
- существующие спортивные приложения;
- Apple Health;
- Google Health Connect;
- спортивные часы;
- датчики пульса;
- датчики кислорода;
- Push Notification Provider.

Требования:

- все интеграции проходят через Integration Service или API Gateway;
- внешние сервисы не получают прямой доступ к внутренним сервисам;
- ошибки внешних интеграций не должны блокировать основные сценарии;
- для внешних API используются timeout, retry и fallback.

---

# NFR-09. Развёртывание

Система должна поддерживать безопасный выпуск новых версий.

Требования:

- zero-downtime deployment;
- CI/CD pipeline;
- автоматические тесты;
- автоматические security checks;
- blue-green или canary deployment;
- возможность быстрого rollback;
- хранение инфраструктуры в виде кода.

---

# NFR-10. Совместимость

Пользовательские приложения должны работать на основных мобильных платформах.

Требования:

- поддержка iOS;
- поддержка Android;
- поддержка разных регионов;
- локализация интерфейса;
- корректная работа при нестабильном мобильном интернете.

---

# NFR-11. Работа с данными

Требования к хранению данных:

- каждый сервис владеет своей базой данных;
- прямой доступ к базе другого сервиса запрещён;
- обмен данными выполняется через API или события;
- тренировочные события сохраняются в Kafka;
- аналитические данные выгружаются в Data Lake;
- медиафайлы хранятся в Object Storage.

---

# NFR-12. Эксплуатация

Система должна быть удобной для сопровождения.

Требования:

- единые стандарты логирования;
- единые health-check endpoints;
- readiness и liveness probes;
- runbooks для критичных инцидентов;
- мониторинг стоимости облачных ресурсов;
- автоматическое масштабирование;
- регулярное тестирование восстановления.

---

# Приоритет НФТ

| Приоритет | Нефункциональные требования |
|----------|-----------------------------|
| Высокий | Безопасность, доступность, масштабируемость, отказоустойчивость |
| Средний | Производительность, наблюдаемость, развёртывание |
| Низкий | Локализация, расширенная совместимость |

---

# Влияние НФТ на архитектуру

| НФТ | Архитектурное решение |
|----|------------------------|
| Доступность | Kubernetes, репликация сервисов и БД |
| Масштабируемость | Stateless-сервисы, HPA, Kafka |
| Безопасность | OAuth2/OIDC, JWT, mTLS, RBAC |
| Производительность | Redis, кэширование, асинхронная обработка |
| Наблюдаемость | Prometheus, Grafana, Loki, Tempo |
| Отказоустойчивость | Circuit Breaker, Retry, Timeout |
| Интеграции | Integration Service, API Gateway |
