# ADR-001. Выбор микросервисной архитектуры

## Статус

Согласовано

## Контекст

Платформа должна поддерживать большое количество функций: тренировки, социальные группы, рекомендации, акции, соревнования, интеграции с устройствами и интернет-магазином.

Компания имеет большой штат разработчиков, поэтому разные команды должны иметь возможность независимо развивать отдельные части системы.

## Решение

Использовать микросервисную архитектуру с разделением по бизнес-доменам.

Основные домены:

- Identity;
- User Profile;
- Training;
- Social;
- Search;
- Recommendation;
- Equipment;
- Gamification;
- Notification;
- Promotion;
- Analytics;
- Integration.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Монолит | Не подходит для глобальной платформы, плохо масштабируется и усложняет параллельную разработку |
| Модульный монолит | Лучше классического монолита, но масштабируется только целиком |

## Последствия

Плюсы:

- независимая разработка сервисов;
- независимое масштабирование;
- изоляция отказов;
- проще развивать систему поэтапно.

Минусы:

- выше сложность эксплуатации;
- требуется централизованный мониторинг;
- сложнее тестирование интеграций.
- требуется больше ресурсов

# ADR-002. Использование API Gateway

## Статус

Согласовано

## Контекст

К системе обращаются мобильные приложения, административная панель, маркетинг и внешние интеграции. Прямой доступ клиентов к микросервисам усложняет безопасность и поддержку API.

## Решение

Использовать API Gateway как единую точку входа.

API Gateway отвечает за:

- маршрутизацию запросов;
- проверку JWT (json web token);
- базовую авторизацию;
- rate limiting;
- защиту публичного API;
- логирование входящих запросов.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Прямой доступ к сервисам | Повышает связанность клиентов с внутренней архитектурой |
| Отдельный gateway для каждого домена на старте | Усложняет MVP и эксплуатацию |

## Последствия

Плюсы:

- централизованная защита API;
- упрощение клиентских приложений;
- единая точка контроля трафика.

Минусы:

- API Gateway становится критичным компонентом;
- требуется отказоустойчивое развертывание.

# ADR-003. Использование событийной архитектуры

## Статус

Согласовано

## Контекст

Система должна обрабатывать большое количество событий: завершение тренировок, достижения, уведомления, обновление рейтингов, аналитика и рекомендации.

Синхронная обработка всех действий ухудшит производительность и повысит связанность сервисов.

## Решение

Использовать Event Driven Architecture для фоновых и высоконагруженных процессов.

Примеры событий:

- WorkoutStarted;
- WorkoutCompleted;
- TrainingMetricsUpdated;
- AchievementUnlocked;
- EquipmentUsageUpdated;
- PromotionActivated;
- UserJoinedCompetition.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Только REST | Повышает связанность и ухудшает устойчивость при пиковых нагрузках |
| Batch-обработка | Не подходит для уведомлений, рейтингов и соревнований почти в реальном времени |

## Последствия

Плюсы:

- слабая связанность сервисов;
- хорошая масштабируемость;
- удобная обработка фоновых задач.

Минусы:

- сложнее отладка;
- требуется мониторинг очередей и lag;
- появляется eventual consistency.

# ADR-004. Использование Kafka

## Статус

Согласовано

## Контекст

Платформа должна выдерживать большое количество событий во время массовых соревнований и потоковую обработку данных тренировок.

## Решение

Использовать Kafka как брокер событий.

Kafka используется для:

- событий тренировок;
- событий устройств;
- событий социальных действий;
- событий достижений;
- событий промоакций;
- передачи данных в аналитику.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| RabbitMQ | Хорош для очередей задач, но хуже подходит для потоковой аналитики |
| Cloud Pub/Sub | Удобен, но усиливает зависимость от конкретного провайдера |
| REST callbacks | Повышают связанность сервисов |

## Последствия

Плюсы:

- высокая пропускная способность;
- хранение истории событий;
- возможность повторной обработки;
- хорошая база для аналитики.

Минусы:

- требуется отдельная эксплуатация Kafka;
- нужна схема событий и контроль совместимости.

# ADR-005. Использование Kubernetes

## Статус

Согласовано

## Контекст

Система состоит из большого количества микросервисов и должна масштабироваться горизонтально. Также компания использует нескольких облачных провайдеров, поэтому важно снизить зависимость от конкретного облака.

## Решение

Разворачивать сервисы в Kubernetes.

Kubernetes используется для:

- запуска микросервисов;
- горизонтального масштабирования;
- автоматического восстановления контейнеров;
- rolling updates;
- изоляции окружений;
- переносимости между облачными провайдерами.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Виртуальные машины | Сложнее масштабировать и сопровождать большое количество сервисов |
| Serverless | Не все сервисы подходят под serverless-модель, есть риск vendor lock-in |
| Managed PaaS | Удобно, но повышает зависимость от конкретного облака |

## Последствия

Плюсы:

- стандартная платформа для микросервисов;
- автоматическое масштабирование;
- переносимость между облаками.

Минусы:

- требуется компетенция эксплуатации Kubernetes;
- выше сложность инфраструктуры.

# ADR-006. Использование Database per Service

## Статус

Согласовано

## Контекст

Микросервисы должны быть независимыми. Использование общей базы данных приведёт к сильной связанности сервисов и затруднит изменения схемы данных.

## Решение

Каждый сервис владеет своей базой данных.

Примеры:

- User DB;
- Training DB;
- Social DB;
- Equipment DB;
- Gamification DB;
- Promotion DB;
- Recommendation DB.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Общая база данных | Высокая связанность сервисов |
| Общая схема с разными таблицами | Формально разделяет данные, но не даёт настоящей независимости |

## Последствия

Плюсы:

- независимое развитие сервисов;
- отсутствие прямой зависимости от схемы чужой БД;
- проще масштабировать домены отдельно.

Минусы:

- сложнее строить сквозные отчёты;
- требуется событийная синхронизация;
- возможна eventual consistency.

# ADR-007. Использование OAuth2 / OpenID Connect и JWT

## Статус

Согласовано

## Контекст

Система работает с персональными данными пользователей и должна поддерживать безопасный вход, внешних провайдеров аутентификации и разграничение доступа.

## Решение

Использовать OAuth2 / OpenID Connect для аутентификации и JWT для передачи сведений о пользователе между API Gateway и сервисами.

Identity Service отвечает за:

- вход пользователя;
- интеграцию с внешними Identity Provider;
- выпуск JWT;
- управление сессиями;
- управление ролями.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Собственная простая авторизация | Высокие риски безопасности |
| Session-based auth без токенов | Хуже подходит для мобильных клиентов и распределённой архитектуры |
| Только API keys | Не подходит для пользовательских сценариев |

## Последствия

Плюсы:

- стандартный подход к аутентификации;
- поддержка внешних провайдеров;
- удобная передача user context в сервисы.

Минусы:

- требуется управление сроком жизни токенов;
- нужна защита refresh token;
- бизнес-авторизацию всё равно должны проверять доменные сервисы.

# ADR-008. Использование mTLS для межсервисного взаимодействия

## Статус

Согласовано

## Контекст

Внутри платформы много сервисов, которые обмениваются чувствительными данными. Недостаточно проверять только пользователя, нужно также проверять сервис, который выполняет запрос.

## Решение

Использовать mTLS для межсервисного взаимодействия.

Применяется для:

- REST/gRPC вызовов между сервисами;
- подключения сервисов к Kafka;
- защиты внутренних API.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Только внутренняя сеть Kubernetes | Недостаточная защита при компрометации пода или сети |
| Только JWT | JWT идентифицирует пользователя, но не сам сервис |
| API keys между сервисами | Сложнее безопасно управлять ключами |

## Последствия

Плюсы:

- сервисы взаимно проверяют друг друга;
- снижается риск несанкционированного доступа;
- лучше соответствует Zero Trust.

Минусы:

- требуется управление сертификатами;
- усложняется эксплуатация;
- желательно внедрение service mesh или централизованного управления сертификатами.

# ADR-009. Использование Redis для кэширования

## Статус

Согласовано

## Контекст

Некоторые данные читаются часто: профили, рейтинги, рекомендации, результаты поиска. Постоянное обращение к БД увеличит задержки и нагрузку.

## Решение

Использовать Redis для кэширования часто используемых данных.

Кэшируются:

- профили пользователей;
- поисковые запросы;
- рейтинги соревнований;
- персональные рекомендации.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Без кэша | Высокая нагрузка на БД |
| In-memory cache в сервисах | Не подходит при горизонтальном масштабировании |
| CDN | Подходит для статического контента, но не для персональных данных |

## Последствия

Плюсы:

- снижение задержек;
- снижение нагрузки на БД;
- быстрый доступ к рейтингам и рекомендациям.

Минусы:

- требуется стратегия инвалидации;
- возможны устаревшие данные;
- Redis становится важным инфраструктурным компонентом.

# ADR-010. Использование мультирегионального размещения

## Статус

Согласовано

## Контекст

Пользователи находятся по всему миру. Приложение должно быть доступно в разных странах, включая ситуации, когда пользователь временно находится в другом регионе.

Также система должна выдерживать международные соревнования с большим количеством участников.

## Решение

Использовать Multi-region размещение.

Основные элементы:

- Global Load Balancer;
- CDN;
- Kubernetes-кластеры в нескольких регионах;
- репликация критичных данных;
- региональное масштабирование сервисов.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Один регион | Высокие задержки и единая точка отказа на уровне региона |
| Active-passive только для DR | Не решает проблему задержек для пользователей в других странах |
| Полный multi-cloud active-active на старте | Слишком высокая сложность для первого релиза |

## Последствия

Плюсы:

- ниже задержки для пользователей из разных стран;
- выше доступность;
- лучше готовность к международным соревнованиям.

Минусы:

- сложнее репликация данных;
- выше стоимость инфраструктуры;
- требуется продуманная стратегия маршрутизации пользователей.

# ADR-011. Использование централизованной Observability

## Статус

Согласовано

## Контекст

Микросервисная архитектура усложняет поиск причин инцидентов. Для эксплуатации нужны логи, метрики и трассировка.

## Решение

Использовать централизованную Observability-платформу.

Инструменты:

- Prometheus для метрик;
- Grafana для dashboards;
- Loki для логов;
- Tempo для распределённой трассировки;
- Alertmanager для уведомлений об инцидентах.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Логи только внутри сервисов | Невозможно быстро расследовать инциденты |
| Только инфраструктурный мониторинг | Не показывает состояние бизнес-сценариев |
| Разные инструменты в разных командах | Усложняет эксплуатацию и поддержку |

## Последствия

Плюсы:

- быстрый поиск причин ошибок;
- единые dashboards;
- возможность контролировать бизнес-метрики.

Минусы:

- дополнительные расходы на хранение логов и трассировок;
- требуется настройка стандартов логирования.

# ADR-012. Использование Integration Service

## Статус

Согласовано

## Контекст

Платформа должна интегрироваться со спортивными устройствами, фитнес-функциями телефона, существующими приложениями компании и интернет-магазином.

Если каждый сервис будет интегрироваться с внешними системами самостоятельно, архитектура станет сильно связанной и сложной в сопровождении.

## Решение

Использовать Integration Service как отдельный слой интеграции с внешними системами.

Integration Service отвечает за:

- подключение спортивных устройств;
- получение данных из Apple Health / Google Health Connect;
- интеграцию с существующими спортивными приложениями компании;
- нормализацию внешних данных;
- публикацию событий устройств в Kafka.

## Альтернативы

| Альтернатива | Причина отказа |
|-------------|----------------|
| Прямые интеграции из каждого сервиса | Дублирование логики и высокая связанность |
| Интеграция только через API Gateway | Gateway не должен содержать бизнес-логику интеграций |
| Отдельный сервис на каждого поставщика устройств | Усложняет архитектуру на старте |

## Последствия

Плюсы:

- единая точка работы с внешними системами;
- проще добавлять новые устройства;
- доменные сервисы не зависят напрямую от внешних API.

Минусы:

- Integration Service становится важным компонентом;
- требуется обработка ошибок внешних API;
- нужна нормализация данных от разных поставщиков.
